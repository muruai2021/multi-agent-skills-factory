# Multi-Agent Skill Factory 流水线详解（主编调度手册）

> 本文档是 [SKILL.md](SKILL.md) 的完整流水线细节，供主编调度时参考。

---

## 流水线概览

```
用户说"封装Skill"
         ↓
主编接收任务，解析SOP
         ↓
   ┌─────┴─────┐
   ↓           ↓
研究员         设计师
分析SOP       产出设计规格书
   ↓           ↓
   └─────┬─────┘
         ↓
开发者编写SKILL.md
         ↓
测试员执行测试用例
         ↓
评审员量化打分
         ↓
   ┌────┴────┐
   ↓         ↓
✅ ≥75分   ❌ <75分
注册上线    返回开发者修改
               ↑
         （循环直到达标）
```

---

## 主编的7步调度法

### Step 1: 接收并解析SOP

主编收到用户SOP描述后，首先解析：

| 问题 | 产出 |
|------|------|
| 技能叫什么？（中文名 + skill-xxx） | 技能命名 |
| 它解决什么重复性工作？ | 核心价值 |
| 谁会使用这个技能？ | 目标用户 |
| 触发词是什么？ | 触发条件 |
| 成功的标准是什么？ | 验收条件 |

### Step 2: 并行调度研究员 + 设计师

**并行执行**（节省50%时间）：

```
研究员任务：
"分析以下SOP，产出SOP分析报告：
[SOP内容]"

设计师任务：
"根据研究员产出的SOP分析报告，
产出技能设计规格书"
```

### Step 3: 主编汇总，产出技能设计规格书

研究员和设计师完成后，主编汇总产出《技能设计规格书》：

```markdown
## 技能设计规格书

### 基本信息
- **技能名称**：[中文名]
- **skill名称**：skill-xxx
- **版本**：1.0.0
- **业务域**：[部门/流程]

### 触发词
**必须满足所有条件才触发：**
1. [条件列表]

**满足任一条件不触发：**
1. [排除条件]

### 核心流程
#### 流程1：[子流程名称]
**触发条件**：[条件]
**执行步骤**：
1. [步骤1]
2. [步骤2]
3. [步骤3]

#### 流程2：[子流程名称]
...

### 输入输出
- **输入**：[描述]
- **输出**：[描述]

### 错误处理
- [异常场景1] → [处理方式]
- [异常场景2] → [处理方式]

### 边界场景
- [场景1]：[处理方式]
- [场景2]：[处理方式]
```

### Step 4: 调度开发者编写SKILL.md

主编给开发者下达任务：

```markdown
## 开发任务

请根据以下技能设计规格书，编写完整的SKILL.md：

### 技能设计规格书
[规格书内容]

### 要求
1. 遵循 hermes-agent-skill-authoring 标准
2. frontmatter必须包含：name, description, version, author, license, platforms, metadata.hermes
3. description ≤ 1024字符，以"Use when"开头
4. references/目录存放 >8KB 的内容
5. 测试文件命名为 test_pool.md（非 test_cases.md）
6. 代码注释使用中文

### 输出路径
~/.hermes/skills/[category]/[skill-name]/SKILL.md
```

### Step 5: 调度测试员执行测试用例

开发者完成后，主编给测试员下达任务：

```markdown
## 测试任务

请对以下SKILL.md执行完整测试：

### 待测文件
[SKILL.md路径]

### 测试要求
1. 语法验证：frontmatter格式、description长度（≤1024字符）、以"Use when"开头
2. 功能测试：核心流程是否按预期工作
3. 边界测试：极端输入的处理
4. 安全测试：敏感信息、注入攻击

### 输出
测试报告，包含：通过/未通过 + 具体问题列表
```

### Step 6: 调度评审员量化打分

测试通过后，主编给评审员下达任务：

```markdown
## 评审任务

请对以下SKILL.md进行五维打分：

### 待评文件
[SKILL.md路径]

### 评估维度（每项20分，满分100）
1. 完整性（25%）：核心流程是否覆盖完整SOP
2. 正确性（25%）：输出是否符合hermes-agent-skill-authoring标准
3. 易用性（20%）：触发词是否清晰、使用是否简单
4. 安全性（20%）：是否有敏感信息泄露风险
5. 可维护性（10%）：代码结构是否清晰、references/是否规范

### 评分标准
- 90-100分：✅ 优秀，直接上线
- 75-89分：⚠️ 良好，小修小补后上线
- <75分：🔧 需要修改，退回开发者

### 输出
评审报告，包含：各项得分 + 总分 + 改进建议
```

### Step 7: 主编决策（循环 or 交付）

根据评审分数决策：

| 分数 | 主编决策 |
|------|----------|
| ≥90分 | ✅ 直接注册上线，输出技能路径 |
| 75-89分 | ⚠️ 将改进建议传给开发者，修改后重新评审 |
| <75分 | 🔧 将问题列表传给开发者，循环修改（最多6轮） |
| 达到6轮 | ⚠️ 交付当前最高分版本，说明未达标原因 |

---

## 迭代循环控制

```
第1轮：开发 → 测试 → 评审（评分：X）
         ↓
      X < 75？
         ↓ 是
第2轮：开发 → 测试 → 评审（评分：Y）
         ↓
      Y < 75？
         ↓ 是
...（最多6轮）
         ↓
第6轮：开发 → 测试 → 评审（评分：Z）
         ↓
      Z < 75？
         ↓ 是
交付Z分版本，说明剩余改进空间
```

---

## 注册上线

评审≥75分后，主编完成注册：

1. 更新 `~/.hermes/skills/registry.yaml`
2. 更新技能 `metadata.yaml` 的 `status: production`
3. 告知用户技能路径和使用方式

---

## 完整目录结构

```
~/.hermes/skills/
├── registry.yaml           # 全局技能注册表
└── {category}/           # 按业务域分类
    └── {skill-name}/
        ├── SKILL.md       # 技能核心定义
        ├── config.yaml    # 运行时配置（可选）
        ├── metadata.yaml  # 元数据（可选）
        └── references/   # 支撑文档
            ├── SOP.md
            ├── test_pool.md
            ├── design-spec.md
            └── eval-dimensions.md
```

## 执行示例

**场景：** 把"客户投诉自动处理"封装为Skill

```
用户：把这个投诉处理SOP封装成Skill

① 主编解析：
   技能名：skill-complaint-handler
   触发词：收到客户投诉

② 并行：研究员分析 + 设计师设计

③ 开发者编写 SKILL.md

④ 测试员执行测试用例
   TC-001 ✅ | TC-002 ✅ | BC-001 ✅

⑤ 评审员打分：88分

⑥ ✅ ≥75分，注册上线

输出：~/.hermes/skills/productivity/skill-complaint-handler/
```
