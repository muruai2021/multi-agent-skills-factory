# 评估维度与评分标准

> 评审员Agent使用此文档对Skill进行五维打分。
>
> **权重（与 pipeline-multi-agent.md Step 6 完全一致）：**
> 完整性25% · 正确性25% · 易用性20% · 安全性20% · 可维护性10%
>
> **评分计算：**
> `总分 = 完整性×0.25 + 正确性×0.25 + 易用性×0.20 + 安全性×0.20 + 可维护性×0.10`

---

## 五维评估体系

### 得分等级换算（每项满分20分）

| 得分 | 等级 | 说明 |
|------|------|------|
| 18–20 | 优秀 | 超越预期，结构优雅 |
| 14–17 | 良好 | 符合标准，能用 |
| 10–13 | 及格 | 基本达标，有缺陷 |
| 0–9 | 不及格 | 严重缺失，无法使用 |

### 各维度详细权重

| 维度 | 权重 | 满分 | 优秀(18-20) | 良好(14-17) | 及格(10-13) | 不及格(0-9) |
|------|------|------|-------------|-------------|-------------|-------------|
| **完整性** | 25% | 20 | 完美覆盖SOP，含边界和异常 | 基本覆盖核心SOP | 部分缺失 | 缺关键步骤 |
| **正确性** | 25% | 20 | 超越预期，结构优雅 | 符合标准 | 大多正确 | 输出错误 |
| **易用性** | 20% | 20 | 一用就会，触发精准 | 较清晰 | 需说明 | 触发模糊 |
| **安全性** | 20% | 20 | 零风险 | 基本安全 | 需加固 | 有泄露风险 |
| **可维护性** | 10% | 20 | 非常优雅 | 较清晰 | 结构松散 | 混乱难改 |

## 各维度详细评估标准与扣分规则

### 1. 完整性（权重25%，满分20）

**评估内容：**
- frontmatter字段是否完整（name, description, version, author, license, platforms, metadata.hermes）
- 触发词是否覆盖所有应触发场景
- 核心流程是否覆盖完整SOP
- 是否包含错误处理机制
- 是否包含边界场景处理

**扣分项：**
- 缺frontmatter字段 → -5分/项
- 触发词遗漏 → -3分/场景
- 流程步骤缺失 → -5分/步
- 无错误处理 → -5分
- 无边界场景 → -3分

### 2. 正确性（权重25%，满分20）

**评估内容：**
- SKILL.md格式是否符合 hermes-agent-skill-authoring 标准
- description是否以"Use when"开头
- description是否≤1024字符
- 流程描述是否准确无歧义
- 文件路径和引用是否正确

**扣分项：**
- description不以"Use when"开头 → -5分（直接扣5）
- description > 1024字符 → -3分
- 格式不符合标准 → -5分/项
- 路径/引用错误 → -3分/处

### 3. 易用性（权重20%，满分20）

**评估内容：**
- 触发词是否简洁明了（多种自然语言表达）
- 使用说明是否清晰
- 输入输出定义是否明确
- 是否有示例

**扣分项：**
- 触发词模糊/歧义 → -5分
- 无使用示例 → -3分
- 输入输出不明确 → -3分/处

### 4. 安全性（权重20%，满分20）

**评估内容：**
- 是否有敏感信息泄露（如密码、token、密钥）
- 是否有注入攻击风险
- 是否有越权操作可能
- 是否有数据泄露风险

**扣分项：**
- 含敏感信息 → -10分（直接不及格）
- 注入风险 → -5分/处
- 越权可能 → -5分/处

### 5. 可维护性（权重10%，满分20）

**评估内容：**
- 代码结构是否清晰
- references/目录是否规范
- 测试文件命名是否正确（test_pool.md）
- 文档是否完整

**扣分项：**
- 无references/但内容>8KB → -3分
- test文件命名错误 → -3分
- 文档不完整 → -2分/处

---

### 6. 自我迭代能力（权重 — 加分项，最高+10）

**评估内容：**
- references/failure_case_log.md 是否存在且记录规范
- references/self-review-template.md 是否存在
- 是否有 ≥ 3 条已修复案例（说明迭代机制在运作）
- 是否有定期复盘记录

**加分规则：**
- 有错误日志且 ≥ 3 条记录 → +3分
- 有自我复盘报告且有改进措施 → +3分
- 已修复案例进入回归测试池 → +2分
- 最近一次使用距今 ≤ 30 天无新增错误 → +2分

**评分计算：**
```
总分 = 完整性×0.25 + 正确性×0.25 + 易用性×0.20 + 安全性×0.20 + 可维护性×0.10 + 自我迭代加分
（上限100分）
```

## 评分等级

| 等级 | 分数范围 | 处理方式 |
|------|----------|----------|
| 🏆 优秀 | 90–100 | ✅ 直接上线 |
| ⚠️ 良好 | 75–89 | 小修小补后上线 |
| 🔧 需修改 | 60–74 | 退回开发者修改，最多6轮 |
| ❌ 不合格 | <60 | 重做 |

## grading.json 输出规范（关键）

评审员必须同时输出 grading.json，格式如下：

```json
{
  "skill_name": "skill-xxx",
  "reviewer": "评审员Agent",
  "timestamp": "YYYY-MM-DD HH:MM:SS",
  "dimensions": [
    {
      "name": "完整性",
      "score": 17,
      "max": 20,
      "weight": 0.25,
      "evidence": "覆盖核心SOP，但边界场景缺失2个",
      "issues": ["缺少 X 场景处理", "Y 步骤描述模糊"]
    },
    {
      "name": "正确性",
      "score": 18,
      "max": 20,
      "weight": 0.25,
      "evidence": "符合 hermes-agent-skill-authoring 标准",
      "issues": []
    },
    {
      "name": "易用性",
      "score": 15,
      "max": 20,
      "weight": 0.20,
      "evidence": "触发词较清晰，但缺少使用示例",
      "issues": ["建议增加 1-2 个使用示例"]
    },
    {
      "name": "安全性",
      "score": 20,
      "max": 20,
      "weight": 0.20,
      "evidence": "无敏感信息泄露，无注入风险",
      "issues": []
    },
    {
      "name": "可维护性",
      "score": 16,
      "max": 20,
      "weight": 0.10,
      "evidence": "目录结构规范，references/组织良好",
      "issues": ["test_pool.md 建议补充边界测试用例"]
    }
  ],
  "total_score": 86,
  "grade": "良好",
  "pass": true,
  "recommendation": "小修小补后上线：建议增加使用示例。"
}
```

**字段说明：**
- `name`：维度名称（固定5个）
- `score`：得分（0-20整数）
- `max`：满分（固定20）
- `weight`：权重（固定 0.25/0.25/0.20/0.20/0.10）
- `evidence`：得分理由（必填，简短）
- `issues`：问题列表（必填，可为空列表）
- `total_score`：总分（自动计算）
- `pass`：`true` if total_score ≥ 75，else `false`
- `grade`：等级（优秀/良好/需修改/不合格）
